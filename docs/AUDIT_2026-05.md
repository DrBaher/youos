# YouOS Audit — Roadmap to an Accurate, Autonomous Email Assistant

_Generated 2026-05-29 from a 55-agent audit workflow (every finding adversarially re-verified against the code) plus a direct read of the live baheros production state. 48 findings: 40 confirmed, 8 partial (with corrections inline). 0 refuted._

## Implemented in the 2026-05-29 hardening pass (branch `audit/autonomy-hardening`)

All of Tier 0 plus the highest-value bounded Tier 1–3 items are done and tested (full suite green; +14 tests):

- **Tier 0 (all):** push_to_gmail idempotency + atomic claim (no duplicate Gmail drafts); agent DB connections use the tuned `bootstrap.connect` (busy_timeout/WAL); failed sweeps always log an audit row + scheduler alerts on the first failure transition; per-account sweep lock (no cap-exceeding overlap); label-sync dismisses *all* rows for a thread; `GET /api/agent/pending/{id}`; `agent.threshold` whitelisted (float, clamped); auth middleware re-reads config per request (a PIN set after startup now takes effect); gws write uses per-account credentials (no cross-account leak).
- **Tier 1:** classifier scores the *new* content only (quoted history + signature stripped; trivial acks penalized) — kills the biggest thread-reply false-positive class; nightly golden-eval now scores *real* drafts (was scoring blank strings) and is instance-aware.
- **Tier 2/3:** `POST /api/agent/pending/{id}/regenerate` (re-draft in voice with a free-form instruction); agent heartbeat (`last_sweep_at` / `seconds_since_last_sweep` + staleness hint); `doctor` now checks gog *auth validity*, not just that the binary exists; the drafting model is surfaced in `/api/agent/observability`.
- **Tiered auto-push (b67):** opt-in `agent.auto_push.*` — after a sweep, auto-create a Gmail **Draft** (never sends) for high-confidence replies to known, whitelisted senders. Guardrails: disabled + **dry-run by default**, required sender whitelist, confidence floor (0.85, clamped 0.6–1.0), min prior reply pairs, per-day cap. Reuses the idempotent push path; failure-isolated; outcomes surfaced on `TriageResult.auto_pushed`.
- **Follow-up tracking (b68):** `app/agent/followups.py` — owed-inbound (aging pending rows) + awaiting-reply (sent rows with no newer thread activity); `GET /api/agent/followups` + digest fields (`owed_count` / `awaiting_count` + previews). Read-only, no egress.
- **Thread context into generation (b69):** `InboxMessage.thread_history` (from the thread `fetch_unread` already pulls) → `DraftRequest.thread_history` → `generate_draft` prefers it over the regex `From:` extraction, feeding the `[THREAD HISTORY]…[CURRENT MESSAGE]` block. The agent no longer drafts blind on multi-turn threads.

**Still staged** (need the live model/training env, or are larger new subsystems): adapter regression gate; autoresearch composite sensitivity (empirical tuning); calendar proposals; voice-match gating at generation; triage precision/recall harness; structured standing-instruction rules; VIP routing / thread summaries; proactive webhook push.

---

## Thesis

YouOS is ~90% built as an autonomous email assistant — but the last 10%, the parts that make it *trustworthy* and *self-improving*, are dark, inert, or unmeasured. The fastest path to "accurate, useful, autonomous" is to make the loop you already built actually bite, not to add features.

## Live-instance facts (from the baheros DB + nightly.log, not code-reading)

- **The autonomous agent has never run on production.** `agent_pending_drafts`/`agent_audit` tables don't exist in the baheros DB; there is no `agent:` config section (so `agent.enabled` defaults False). The flagship feature is off.
- **Self-optimization keeps 0 improvements every night.** `composite 0.42 → 0.42` on all 14 mutations. Two root causes: (a) the nightly golden-eval scores **blank drafts** (`run_golden_eval()` with no `generate_fn`, `nightly_pipeline.py:491`); (b) the composite is quantized too coarsely — with 44 cases at a 0.02 improvement bar, one case flipping moves composite by only `(1/44)·0.5 = 0.011`, below the bar — so it can never register incremental progress. `scorer.py:160,182`.
- **The feedback loop isn't fed.** `auto_feedback: captured 0 pairs`, export skipped ("need 5"). The 46,973 historical feedback pairs aren't growing.
- **Wiring that the old memory flagged as broken is now FIXED:** the nightly pipeline is scheduled (launchd `com.youos.baheros-nightly.plist`, 01:00) and instance-aware (`YOUOS_INSTANCE_DIR`). The remaining problems are inertness and measurement, not wiring.

## Tier 0 — Correctness & safety bugs (blocks trusting it unattended)

### [draft-quality] Autonomous agent drafts replies without prior thread context
`impact 5/5 · effort M · **BUG**`

- **Now:** fetch_unread (app/agent/inbox_fetch.py:88-121) returns only messages[-1] — the latest message per thread — and discards earlier exchanges, even though it already called get_thread and has thread['messages']. triage.py:174-191 then passes only msg.body to generate_draft. Inside generate_draft, clean_inbound = strip_quoted_text(...) (service.py:1628) truncates at the FIRST quote boundary, removing any inline quoted history. _has_thread_context then checks clean_inbound.count('From:') >= 2 (service.py:396-398), which is almost always false after stripping. So the multi-turn machinery (_extract_thread_parts/_format_thread_context) is effectively dead for the agent path.
- **Gap:** In an ongoing thread (the most common reply situation), the model sees a single message with no conversation history and pattern-matches on style — it can confidently answer the wrong question, miss what was already agreed, or re-ask something already settled. The audit's focus question 'does generation know the full thread context' is answered 'no' for the autonomous path. The data is available (get_thread already fetched it) but thrown away.
- **Fix:** Have fetch_unread keep the last N messages of each thread on InboxMessage (e.g. a thread_history field) and have triage pass them through. In generate_draft, prefer this structured history over the regex-based _extract_thread_parts (which is brittle — it parses 'From:' lines out of a single body). Feed prior turns into _format_thread_context BEFORE strip_quoted_text eats them. Cap by token budget (already have the knapsack at service.py:1809).
- **Risk:** Larger prompts raise the chance of the small LoRA losing the actual question among history; mitigate by clearly delimiting [CURRENT MESSAGE] (already done) and capping history to last 2 exchanges. No trust-boundary change: still draft-only, still local.
- **Evidence:** app/agent/inbox_fetch.py:88-121 (messages[-1] only); app/agent/triage.py:174-191 (passes msg.body only); app/generation/service.py:1628 then 1636 (strip_quoted_text before _has_thread_context); service.py:396-398,401-441 (thread machinery keyed on surviving 'From:' blocks)
- **⚠ Verifier correction:** Substance fully confirmed; two minor precision notes. (a) The detection-killing mechanism in the common Gmail case is that Gmail quotes with 'On ... wrote:' (a strip boundary), not literal 'From:' lines — _has_thread_context keys on 'From:' but strip removes the whole quoted block regardless, so the net 'machinery is dead' conclusion holds. (b) is_bug=true is slightly strong: there is no crash or data corruption and a draft is still produced; this is better characterized as a draft-quality/correctness-of-content gap (plausible-but-possibly-wrong reply) than a hard runtime bug. The fix shape the auditor proposes is sound, but note generate_draft re-strips internally (service.py:1628), so structured history must be threaded through a NEW field and consumed BEFORE that strip call, not just passed in inbound_message.

### [robustness] push_to_gmail has no idempotency/status guard — retries and concurrent calls create duplicate Gmail drafts
`impact 5/5 · effort M · **BUG**`

- **Now:** POST /api/agent/pending/{row_id}/push_to_gmail loads the row, checks only tier and that a draft body exists, then unconditionally calls gmail_write.create_draft() (which the module docstring states is non-idempotent: 'pushes an agent-generated draft into Gmail Drafts as a real reply' — a new draft each call), and only AFTER the write succeeds calls store.mark_sent(...). The route never inspects row['status']. There is no UNIQUE constraint on gmail_draft_id and no SELECT ... WHERE status='pending' precondition.
- **Gap:** Any retry of a slow/failed-looking request, a double-click in /triage, or two near-simultaneous orchestrator calls (Telegram bot + Claude both acting on the digest) will each create a separate real Gmail draft replying to the same inbound. The user finds 2-3 identical draft replies in the same thread. Because create_draft runs before mark_sent, a transient failure in mark_sent leaves the Gmail draft created but the row still 'pending' — the next push re-creates it. This is the single highest-risk path because it is the only one that writes to the user's actual mailbox.
- **Fix:** In push_to_gmail (app/api/agent_routes.py:430): (1) reject up front if row['status'] == 'sent' and row['gmail_draft_id'] is set (return the existing draft id, HTTP 200 idempotent). (2) Make the transition atomic: add a store helper that does UPDATE agent_pending_drafts SET status='sent' WHERE id=? AND status IN ('pending','amended') and treats rowcount==0 as 'already claimed' BEFORE calling create_draft, so a concurrent caller loses the race and returns the existing id instead of writing. (3) Wire app/core/rate_limit.draft_limiter (currently only on /feedback and /stream) onto this route as a backstop.
- **Risk:** Touches the never-send boundary's adjacent surface (real Gmail Drafts). Duplicate drafts erode trust in the autonomous agent and risk the user accidentally sending the same reply twice. The atomic-claim fix must be ordered before the network write or a crash between claim and write could orphan a 'sent' row with no Gmail draft — acceptable (user re-pushes) but should be documented.
- **Evidence:** app/api/agent_routes.py:430-482 (no status check; create_draft then mark_sent); app/ingestion/gmail_write.py:1-8,51-90 (documented non-idempotent, new draft per call); app/agent/store.py:136-148,186-217 (mark_sent just overwrites gmail_draft_id, no precondition); app/db/bootstrap.py:279-283 (gmail_draft_id added with no UNIQUE constraint); app/core/rate_limit.py:57-58 (limiter exists but only imported by feedback_routes/stream_routes)
- **⚠ Verifier correction:** Finding is accurate. One refinement to the proposed fix, not a defect in the finding: the idempotency check should key on gmail_draft_id being already set (an existing Gmail-side draft), NOT on status=='sent' alone — because the separate manual 'Mark sent' route (agent_routes.py:356-358) also sets status='sent' with gmail_draft_id=None, so a status-only guard would wrongly block a manually-marked row from ever being pushed. The atomic claim should be `UPDATE ... SET status='sent' WHERE id=? AND status IN ('pending','amended') AND gmail_draft_id IS NULL` (rowcount==0 => already claimed/pushed). Also worth noting the crash-between-claim-and-write window (orphans a 'sent' row with no draft); the finding already calls this out as acceptable.

### [observability-trust] A totally-failed sweep (gog auth expired, inbox fetch down) writes no audit row, fires no alert, and is invisible
`impact 5/5 · effort M · **BUG**`

- **Now:** run_triage() calls fetch_unread() at triage.py:111 OUTSIDE any try/except. The single agent_audit row is only written near the end (log_sweep at triage.py:290), after fetching and drafting succeed. If fetch_unread raises (expired gog OAuth token, network down, gog binary missing), the exception propagates out of run_triage; the scheduler's _loop catches it at scheduler.py:159-162 and logs at logging.INFO only ('agent loop: sweep for %s failed'). No agent_audit row is appended, no macOS notification fires (the notify_macos call at scheduler.py:164-169 only runs when total_persisted > 0), and the next sweep retries silently. So the most common unattended-failure mode (auth expiry) produces zero user-visible signal anywhere — not in /triage Recent activity, not in the digest, not in stats.
- **Gap:** This is the central trust failure for unattended running. The agent can stop drafting indefinitely and the only symptom is the absence of new drafts, which a user away from their desk will not notice. Because the audit row is never written, sweep_aggregate() success_rate stays at its old healthy value (it only counts sweeps that completed far enough to log), actively masking the outage — the observability card will say '100% successful' while the agent is dead.
- **Fix:** Wrap fetch_unread (and the whole sweep body) so a failed sweep ALWAYS logs an agent_audit row with the error in errors_json and fetched=0 (move the log_sweep into a finally/except path in triage.py, or have the scheduler write a failure audit row when run_triage raises). Have the scheduler track consecutive failures and fire _notify_macos on the first failure transition (not just on persisted>0). Add a digest/observability field for last_failed_sweep so a remote user sees it.
- **Risk:** Writing a failure audit row from the except path must not itself raise (a dead DB would loop). Notification spam on a persistent outage must be debounced (notify once per transition, not every tick). No trust-boundary impact — this is read/observe-only.
- **Evidence:** app/agent/triage.py:111 (fetch_unread, no try/except); app/agent/triage.py:282-310 (log_sweep only on the success path); app/agent/scheduler.py:159-169 (failure swallowed at INFO, notify only on persisted>0)

### [integrations-api] push_to_gmail creates a duplicate Gmail draft on retry — no server-side guard despite docs admitting the hazard
`impact 5/5 · effort S · **BUG**`

- **Now:** POST /api/agent/pending/{id}/push_to_gmail (agent_routes.py:430-482) validates only tier/draft-presence/sender_email, then unconditionally calls create_draft() and mark_sent(). It does NOT check whether the row is already status='sent' or already has a gmail_draft_id. AGENT_OPERATIONS.md:61-66 explicitly documents this as 'NOT idempotent — Creates a new Gmail draft each call. Retrying after a timeout can produce duplicate drafts' and instructs orchestrators to manually GET the row and inspect gmail_draft_id before retrying.
- **Gap:** The server pushes the entire idempotency burden onto every orchestrator's error-handling code. An orchestrator that retries a timed-out push (the common case — push_to_gmail is the one call with a 30s+ Gmail round-trip) will silently create a second draft in the user's Gmail Drafts folder. This is the most write-sensitive endpoint in the whole surface (it's the one place YouOS touches Gmail), and it has the weakest safety. The recommended manual-check workflow also can't be followed because the single-row GET it depends on doesn't exist (see separate finding).
- **Fix:** In push_to_gmail (agent_routes.py:430), short-circuit when the row is already pushed: if row.get('gmail_draft_id'), return 200 with the existing draft id and a pushed_already:true flag instead of calling create_draft again. Optionally accept an Idempotency-Key header and store it on the row so cross-process retries dedupe too. This makes the documented 'check before retry' contract enforced server-side rather than advisory.
- **Risk:** Returning the existing draft id on a re-push is safe and strictly reduces duplicate Gmail drafts. The only behavior change is that a deliberate 're-push after I deleted the Gmail draft' would now no-op; mitigate by allowing a force=true body param. No new trust-boundary exposure — still never sends.
- **Evidence:** app/api/agent_routes.py:443-477 (no status/gmail_draft_id pre-check before create_draft + mark_sent); docs/AGENT_OPERATIONS.md:61-66 (documents the non-idempotency and the duplicate-draft risk)

### [learning-loop] Nightly golden-eval gate scores every case against an EMPTY draft
`impact 4/5 · effort S · **BUG**`

- **Now:** step_golden_eval() in the nightly calls scripts.run_golden_eval.run_golden_eval() with NO arguments (nightly_pipeline.py:491). run_golden_eval defaults generate_fn=None, and when generate_fn is None it sets draft='', detected_mode='unknown' for every case (run_golden_eval.py:104-110). score_case then computes keyword_hit_rate=0 (no keywords match empty text), word_count=0 -> brevity_pass=True, mode_match=False -> status 'fail' for essentially every case. The pipeline reads composite=passed/total and gates on >=0.5 (nightly_pipeline.py:496-499) and records golden_composite in pipeline_last_run.json (lines 858-866). The standalone CLI path (run_golden_eval.main) DOES pass a real generate_fn, so the script looks correct in isolation — only the nightly's in-process call is broken.
- **Gap:** The single quality checkpoint that runs right after fine-tuning is a no-op: it always evaluates blank output, so golden_composite is ~0 every night regardless of how good or bad the new adapter is. Operators watching golden_composite see a meaningless number; any future logic that gates on it is gating on noise. The loop cannot detect whether drafts are improving.
- **Fix:** In nightly_pipeline.step_golden_eval, build the same _generate closure that run_golden_eval.main uses (DraftRequest -> generate_draft) and pass generate_fn, database_url, configs_dir into run_golden_eval(). Resolve database_url/configs_dir from get_settings() so it stays instance-aware. Then golden_composite reflects real drafts and can become a real promotion signal (see regression-gate finding).
- **Risk:** Fixing it makes golden_composite swing from ~0 to a real value; any dashboard alert thresholds tuned to the broken ~0 baseline must be reset. Generating real drafts adds runtime to the nightly. No send-boundary impact — eval only generates drafts in-process, never touches Gmail.
- **Evidence:** scripts/nightly_pipeline.py:491 (run_golden_eval() no args); scripts/run_golden_eval.py:104-110 (generate_fn None -> empty draft), :60-69 (score_case), :150-195 (main builds real _generate); nightly_pipeline.py:496-499, 858-866 (composite read + recorded)
- **⚠ Verifier correction:** Two small nuances, neither weakening the core claim. (a) The finding implies golden_composite only matters to 'any future logic'; in fact step_golden_eval's boolean return already feeds the overall pipeline status via all(steps.values()) at nightly_pipeline.py:931 — today it's always False so status can only be 'partial'/'failed', never 'ok', and once fixed (real composite often <0.5 for early adapters) it will keep flipping status. It is recorded as WARN and does NOT halt the run, so the auditor's 'no fatal gate' framing is right, but it does already affect reported status. (b) The fix must also update tests/test_golden_eval.py:86-90, which currently asserts the broken all-fail behavior, or that test will fail (though that in-process path's empty-draft mode is still legitimately reachable for the dry-run default).

### [robustness] Agent DB connections omit busy_timeout/WAL tuning — concurrent writers hit 'database is locked' immediately
`impact 4/5 · effort S · **BUG**`

- **Now:** store._connect, needs_reply.SenderHistory.count_for, and gmail_label_sync._find_pending_row_for_thread each open raw sqlite3.connect(path) with no timeout and no PRAGMA busy_timeout. bootstrap.connect() (used by the rest of the app and the nightly pipeline) deliberately sets timeout=30s, PRAGMA busy_timeout=30000, and journal_mode=WAL precisely because 'the nightly pipeline runs while the web server is live, so lock contention is normal.' WAL persists per-DB, but busy_timeout is per-connection and defaults to 0 on these agent connections.
- **Gap:** When two writers contend — a scheduled sweep's upsert_pending/log_sweep committing while a manual /triage POST or a push_to_gmail mark_sent commits, or while the nightly LoRA pipeline writes — the agent connection with busy_timeout=0 raises sqlite3.OperationalError: database is locked immediately instead of waiting. In the scheduler loop this surfaces as a swallowed 'sweep failed' log (scheduler.py:159-162) and the sweep silently produces zero drafts; in the API path it becomes a 500. Drafts the agent generated (expensive model calls) are lost for that tick.
- **Fix:** Replace the raw sqlite3.connect calls in app/agent/store.py:36-39, app/agent/needs_reply.py:152, and app/agent/gmail_label_sync.py:175 with the shared app/db/bootstrap.connect() (or a thin wrapper that applies timeout + PRAGMA busy_timeout=30000). This is the same fix the rest of the codebase already standardized on; the agent module just didn't adopt it.
- **Risk:** Silent under-performance of the autonomous loop: the user trusts the agent to sweep every 15 min but a colliding nightly job or manual run quietly drops a sweep's output. No data corruption (WAL is safe), but lost work and an invisible failure mode that undermines 'autonomous'.
- **Evidence:** app/agent/store.py:36-39 (sqlite3.connect, no timeout/pragma); app/agent/needs_reply.py:152 (same); app/agent/gmail_label_sync.py:175 (same); app/db/bootstrap.py:16-28 (the tuned connect() the rest of the app uses, with rationale); app/agent/scheduler.py:159-162 (the error gets swallowed at info level)
- **⚠ Verifier correction:** Two of the three cited 'writers' are actually read-only connections, slightly overstating the contention surface (though the central writer cite is correct and load-bearing): (1) needs_reply.py:152 is a cached SELECT COUNT wrapped in try/except that falls back to n=0, so a lock there is both unlikely (WAL permits concurrent readers; they only block during a checkpoint) and already swallowed. (2) gmail_label_sync.py:175 is a read-only SELECT in _find_pending_row_for_thread; the actual write it leads to (store.mark_dismissed at gmail_label_sync.py:126) goes through store._connect and is therefore already covered by the store.py fix. So the precise, must-fix target is store.py:36-39 (all real agent writes flow through it). Fixing needs_reply/gmail_label_sync read connections is still worthwhile for consistency and read-during-checkpoint robustness, but they are not the immediate write-contention bug.

### [robustness] gws backend writes the draft to the wrong mailbox in multi-account setups (per-account credentials ignored)
`impact 3/5 · effort M · **BUG**`

- **Now:** gmail_write._gws_create_draft builds the gws command and calls subprocess.run(cmd, capture_output=True, text=True, timeout=30) with NO env override. It passes the target mailbox only as --params {"userId": account}. The ingestion read path (adapters.GwsSource._run_json) selects the actual mailbox by setting env['GOOGLE_WORKSPACE_CLI_CREDENTIALS_FILE'] from the ingestion.gws_credentials account→file map; the write path never does this.
- **Gap:** gws is single-account per credentials file — userId in --params does not pick the mailbox, the credentials file does (this is stated verbatim in adapters.py:158-165 and gmail_write.py:193-195). So for a user with two accounts on the gws backend, push_to_gmail for account B will create the draft reply (containing private content from account B's inbound) in whatever mailbox the ambient/default gws credentials point to — frequently account A. A reply meant for one inbox lands, visibly, in another account's Drafts. The native backend keys tokens by account file (correct) and gog passes --account (correct); only gws is broken, so this is a backend-specific correctness bug, not a design limitation.
- **Fix:** In gmail_write._gws_create_draft (app/ingestion/gmail_write.py:172-235), mirror adapters.GwsSource: load the per-account map via adapters._load_gws_credentials(), and if a creds path exists for account, pass env with GOOGLE_WORKSPACE_CLI_CREDENTIALS_FILE set on the subprocess.run call. Better: route the write through a shared GwsSource transport so the credential-selection logic lives in exactly one place (matching the b47 'one invocation site' lesson). Add a doctor check that flags gws backend + multiple user.emails + missing ingestion.gws_credentials entries.
- **Risk:** Cross-account data exposure: an agent-drafted reply quoting/answering a private email from account B materializes in account A's Drafts folder, where a different persona/recipient context applies. Directly violates the multi-account isolation the trust model assumes. Only triggers on the gws backend (default is gog), which bounds the blast radius but makes it easy to miss in testing.
- **Evidence:** app/ingestion/gmail_write.py:204-211 (subprocess.run with no env=); app/ingestion/gmail_write.py:191-196 (docstring admits credentials file determines mailbox, calls it a should-map TODO); app/ingestion/adapters.py:274-279 (the read path DOES set GOOGLE_WORKSPACE_CLI_CREDENTIALS_FILE per account); app/ingestion/adapters.py:158-165 ('gws is single-account per credential, no per-command --account')
- **⚠ Verifier correction:** Impact lowered from 4 to 3: the blast radius is genuinely narrow. gog is the default (config.py:103-109; youos_config.yaml sets no google_backend key, so it defaults to gog), so the bug only triggers on the non-default gws backend AND multiple accounts AND a missing/partial ingestion.gws_credentials mapping. Because of the never-send trust boundary the misrouted artifact is an un-sent Draft sitting in the wrong mailbox (not an auto-sent email), and a human reviewing Drafts in Gmail would likely notice it's in the wrong account. Serious but narrow → 3 rather than 4. The doctor-check portion of the enhancement is valid and not already implemented.

### [robustness] Scheduled sweep and manual/API triage can run concurrently for the same account — no mutex, daily cap double-counts
`impact 3/5 · effort M · **BUG**`

- **Now:** scheduler._loop runs each account's sweep via run_in_executor(None, ...) on the default thread pool. Meanwhile /api/agent/triage (agent_routes.py:496) and the CLI (cli.py:586) call run_triage synchronously for the same account. There is no lock, in-progress flag, or per-account serialization anywhere (grep for Lock/in_progress in app/agent finds nothing). The daily cap is enforced by reading count_persisted_today then computing cap_remaining = daily_cap - already and decrementing a local counter (triage.py:138-201) — a classic read-then-write with no atomicity.
- **Gap:** If a scheduled tick fires while the user clicks 'Run triage' (a realistic race: the notification just arrived), both sweeps fetch the same unread set, both read the same 'already' count, and both can generate+persist up to cap_remaining drafts — so the account can exceed agent.daily_draft_cap (the explicit runaway-loop guardrail) by up to 2x. The message_id UNIQUE constraint prevents duplicate ROWS for the same inbound, but two sweeps with slightly different unread sets (new mail arrived between fetches) each consume cap budget independently. Wasted model time and a guardrail that doesn't actually bound the day.
- **Fix:** Add a lightweight per-account advisory lock around run_triage: an asyncio.Lock in the scheduler plus a process-level guard (e.g. an agent_sweep_lock row or a SELECT ... claiming a 'sweep_in_progress' marker) so a manual run and a scheduled tick serialize. For the cap specifically, enforce it at INSERT time (e.g. a trigger or a transactional COUNT-and-insert in upsert_pending) rather than a pre-loop read, so concurrent sweeps see each other's writes.
- **Risk:** The daily cap is a safety guardrail against a runaway loop on a noisy inbox; a race that lets it be exceeded weakens that protection. Combined with no rate limit on the triage endpoint, an orchestrator that retries triage on timeout could amplify model load. No mailbox writes occur here (drafts only persist locally), so it does not directly breach the never-send boundary.
- **Evidence:** app/agent/scheduler.py:153-162 (sweeps dispatched with no cross-sweep coordination); app/api/agent_routes.py:496-531 and app/cli.py:586-622 (independent synchronous run_triage entry points); app/agent/triage.py:138-201 (count_persisted_today read, then local cap_remaining decrement — not atomic); app/agent/store.py:277-289 (count is a plain SELECT)

### [robustness] Gmail label-sync dismisses the wrong row when multiple rows share a thread, and a remove-label failure causes repeat dismissals
`impact 3/5 · effort M · **BUG**`

- **Now:** Before each sweep, run_triage calls sync_gmail_label_dismissals (failure-isolated). For a labelled thread it calls _find_pending_row_for_thread, which returns only the HIGHEST-id pending/amended row for that thread (ORDER BY id DESC LIMIT 1), dismisses just that one, then tries to remove the Gmail label. If the dismissal succeeds but _gog_modify_remove_label fails, the code logs a warning and does NOT roll back ('user signalled intent', gmail_label_sync.py:129-138).
- **Gap:** Two real edge cases: (1) If a thread legitimately has multiple actionable rows (the module's own docstring acknowledges this happens when message_id changes across sweeps), only one is dismissed per sync; the others remain pending and will be re-drafted/surfaced, so the user's 'skip this' gesture is partially ignored. (2) When label removal fails (transient gog auth), the label stays on the thread, so the NEXT sweep re-runs the search, finds the same thread, and dismisses the next-highest pending row — quietly burning through the thread's rows one sweep at a time and potentially dismissing a row the user did NOT intend to skip if a new draft was generated for that thread in between.

### [triage-accuracy] Cold-outreach penalty fights the very drafts it's meant to enable
`impact 1/5 · effort S · ⚠ partial`
